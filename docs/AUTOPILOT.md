# Autopilot

## Purpose

This file defines the automatic context and coordination rules that should run without an agent needing to remember them manually.

Autopilot exists to make context discipline habitual instead of optional.

## Automatic Stages

Frontend uses `ctx:auto` as the trigger surface and then fans into `node scripts/dev/context-autopilot.mjs <stage>`.

Supported stages in this repo:

- `safe-status`
- `safe-sync-start`
- `safe-sync-end`
- `pre-push`
- `post-merge`
- `manual`

## What Autopilot Should Do

At the appropriate stages, autopilot should:

- create a provisional checkpoint if one does not exist yet
- create a provisional claim on feature branches when a safe path boundary can be inferred
- create a provisional task contract on feature branches when no branch contract exists yet

Snapshot save/compact behavior remains owned by `ctx:auto`.

## What Autopilot Must Not Do

Autopilot must not:

- invent irreversible project decisions
- auto-merge branches
- auto-push code
- auto-delete canonical docs
- silently overwrite a richer human-written checkpoint with worse autogenerated text
- silently mark a task contract as complete

## Checkpoint Rule

If the branch has no latest checkpoint, autopilot may create a provisional checkpoint using:

- inferred or configured surface
- a fallback objective derived from the branch
- default canonical docs from `context-kit.json`

Humans or agents should later replace the provisional objective with a task-specific one.

## Claim Rule

On feature branches, autopilot may create a provisional claim only when:

- no active branch claim exists
- changed files allow a reasonable owned path prefix
- the inferred path depth meets the configured minimum

If no safe path boundary can be inferred, autopilot should skip claim creation and leave the normal coordination rule in place.

## Task Contract Rule

On feature branches, autopilot may create a provisional task contract only when:

- no active branch contract exists
- a branch-level work item clearly exists
- the goal can be expressed as a placeholder without inventing real product decisions

Humans or agents should replace provisional finish criteria with task-specific finish criteria before treating the task as complete.

## Configuration

Autopilot reads `context-kit.json.autopilot`.

Key fields:

- `enabled`
- `agentId`
- `autoCheckpoint`
- `autoClaim`
- `autoContract`
- `minimumClaimPathDepth`
- `defaultSurface`
- `defaultObjectivePrefix`
- `defaultDocs`

## Environment Overrides

Useful overrides:

- `CTX_AUTOPILOT_DISABLED=1`
- `CTX_SURFACE=<surface>`
- `CTX_OBJECTIVE=<text>`
- `CTX_WORK_ID=<id>`
- `CTX_AUTO_CLAIM_PATH=<path>`
- `CTX_AGENT_ID=<agent>`
- `CTX_CONTRACT_ID=<id>`
- `CTX_ALLOW_INCOMPLETE_CONTRACT=1`

## Merge Discipline

Autopilot is not a substitute for validation.

It exists to reduce missed process steps before:

- `docs:check`
- `ctx:check -- --strict`
- `contract:check`
- `coord:check`
- project `gate/check/build`
- remote CI
